# Service Portal and Employee Center Lessons

- Scoped Service Portal widget server scripts cannot assume OOTB/global helpers are directly visible. Prefer `global.HelperName` when calling global Script Includes such as `KBViewModel`, and keep a fallback only when it is safe. If the helper invokes restricted internals, remove that path instead of fighting scope restrictions.
- Creating `kb_knowledge` demo articles through Table API may leave `workflow_state=draft` even when the payload sends `published`, because Knowledge business logic can override state transitions. For throwaway search test data in the PDI, a constrained Xplore script with `setWorkflow(false)` can set `workflow_state=published`, `published`, `valid_to`, and `active` after creation.
- Do not call APIs that are blocked in scoped applications from widget server scripts, such as `GlideAccessibility.isEnabled()`. Default conservatively or use data already available to the widget.
- Treat OOTB Knowledge view-count behavior with care in scoped widgets. Previewing an article inside a custom widget should not necessarily call `KBViewModel().increaseKBView()` or queue `kb.view`, because those paths can trigger restricted global functions.
- For Service Portal client scripts, use `api.controller = function(...) { ... };` rather than a bare anonymous function. For link scripts, use `function link(scope, element, attrs, controller) { ... };`.
- OOTB Service Portal child widgets can add Bootstrap tooltip attributes such as `title`, `data-toggle="tooltip"`, and `data-original-title`. When a wrapper redesign should not show hover help text, remove those attributes in the wrapper link function after child render, and use a `MutationObserver` if the child widget can rerender.
- When replacing an OOTB search widget with a custom engine, keep record-producer behavior isolated in the cloned widget where possible. Add contextual fallback/synonym behavior there first, so standard portal search remains stable.
- Record-producer contextual search receives natural user sentences, not deliberate search queries. Exact engine matching may work for portal search but still need conservative fallbacks for phrases such as `jobbe hjemmefra` or `configure router`.
- Avoid broad fallback tokens such as `all`, `alle`, and `for`. Require fallback matches to be visible in title, snippet, category, KB name, or number.
- In scoped widgets, avoid keeping AI Search/AISA compatibility code if the custom widget intentionally uses a different search engine. OOTB AISA helpers may be unavailable in scope.
- For widget options, live `sp_instance.widget_parameters` can contain values not exposed in `sp_widget.option_schema`. Verify both when logic depends on `options.<name>`.
- For Topic Content rollup, `content_displayed_from` must use `Current topic only` or `Current topic and all child topics`.
- For portal typography, inspect `sp_theme.css_variables` before changing attached CSS includes. Later Bootstrap/EC variable blocks usually win.
- Service Portal headers (`sp_header_footer`) can carry CSS that overrides theme typography. Remove text typography overrides when the theme should be the master, but preserve icon-font family declarations when pseudo-icons depend on them.
- Employee Center theme CSS includes can bypass `$font-size-base` with compiled global rules such as `body { font-size: 1.6rem; }`. Prefer a theme-specific include or cloned stylesheet when only one portal should change.
- For Employee Center header desktop glyph links, align icon-and-label stacks against avatar/bell controls with the same vertical control box, for example `min-height: 60px` plus flex centering.
- Header badge colors may be widget-local SCSS defaults such as `$ec-header-badge-bg: #FF9E1B !default;`. Define the variable in the theme when color should be brand-controlled; use inline-flex centering for badge layout.
- For branded VĂĄr Energi widgets, put reusable brand tokens such as `$ve-blue`, `$ve-focus`, `$ve-text`, and `$ve-link` in `sp_theme.css_variables`, then let widget CSS declare `!default` fallbacks.
- Blurry-looking VĂĄr Energi portal widget text is often CSS clarity, not font loading. Check old brand variables, hard-coded colors, muted `rgba()`, gradients, heavy shadows, and hard `700`/`800` weights before changing markup.
- Global Service Portal widget edits are captured by the current Global update set preference. Verify target update set before writes; if capture lands in Default, move the latest customer update to the intended update set.
- VĂĄr Energi website-style pill links can use white/default state with dark border/text and black hover/focus state with white text.
- VĂĄr Energi website-style navigation cards can use large soft-blue rectangular tiles with black text and a simple bright-blue right arrow; avoid unnecessary descriptions, counters, and generic glyphs.

## Employee Center To-do Approval Configurations

- Approval To-do records start at `sn_hr_sp_todos_config` such as the OOTB Approvals config, then use related `sn_ex_sp_task_configuration` rows to choose the approval-specific detail layout, and related `sn_ex_sp_task_tab_configuration` rows for tabs.
- `sn_hr_sp.todoPageUtils._getTaskConfigurations()` orders active task configurations by newest `sys_created_on` first. A new specific task configuration can win over older broad configs without changing OOTB records, but deactivate or clean up experimental broad configs so they do not accidentally match unrelated approvals.
- For HR case approvals that need HR-service-specific behavior, prefer a new task config with `table=sysapproval_approver`, `additional_conditions=source_table=sn_hr_core_case`, `reference_column=source_table`, `reference_table=sn_hr_core_case`, and `reference_conditions=hr_service=<sn_hr_core_service sys_id>`. The resolver follows `source_table` plus the `document_id` field to the HR case and applies `reference_conditions` on that referenced record. For concrete child case tables such as `sn_hr_le_case`, `reference_table=sn_hr_core_case` can still match through table hierarchy fallback.
- If the target instance does not offer HR Case tables after choosing `reference_column=source_table`, do not force the configuration. Official Approval Hub documentation positions `source_table` for approvals whose submitted target is a non-task table; HRSD Lifecycle Event approvals often resolve more cleanly through `reference_column=sysapproval`. In that path, keep the task config broad enough for HR case approvals and put HR-service/activity specificity in a custom tab/widget guard or visible script.
- Lifecycle Event Approval activities are the supported place to generate the approval to-do and define approvers/options, but their description is not documented as a per-approval Employee Center body template. For custom approver body copy plus variables, keep the Lifecycle Event Approval activity OOTB and customize the Employee Center Approval Hub task/tab configuration.
- `reference_conditions` are not evaluated when `reference_column` is a normal reference such as `sysapproval`; that path only checks referenced-record table hierarchy. Use the `source_table`/`document_id` path when conditions must run on the approval target record.
- Protected dependent fields such as `sn_ex_sp_task_configuration.reference_column` may be writable on insert but not on update. If a config was created with `reference_column=sysapproval` and needs `source_table`, create a new config instead of trying to patch it in place.
- Tab type choices on `sn_ex_sp_task_tab_configuration` include `activity`, `attachments`, `custom`, `variable_summary`, `variable_editor`, `multiline`, and `selectable_multiline`. Use `type=custom` when a tab needs static explanatory copy plus OOTB content; set `widget` to a Service Portal widget and pass JSON `widget_parameters`, for example `{"table":"sn_hr_core_case","sys_id":"{reference_table.sys_id}"}`.
- To combine static text with Variable Summarizer, create a small wrapper `sp_widget` in the same app scope. In the server script, call `$sp.getWidget('sc-variable-summarizer', { table: options.table, sys_id: options.sys_id || options.sysId, label: <label>, hide_control: true })`, expose it on `data`, and render it with `<sp-widget widget="data.variableSummarizerWidget"></sp-widget>` under the static text. This keeps OOTB variable rendering, MRVS handling, attachment scanning, and `visible_summary` filtering intact.
- When the normal `{reference_table.sys_id}` or table/sys_id handoff fails for HR approval variables, have the wrapper resolve `sysapproval_approver -> sysapproval/document_id -> sn_hr_core_case` itself, then call `new global.GlobalServiceCatalogUtil().getVariablesForTask(hrCase, true)` and pass the resulting visible variable array into `$sp.getWidget('sc-variable-summarizer', { variables: visibleVariables, hide_control: true })`. Passing `variables` directly lets the OOTB summarizer render the same markup while bypassing `TicketConfig`/public-table lookup problems.
- If an embedded `sc-variable-summarizer` still renders blank inside an Employee Center To-do custom tab, render a plain JSON copy of the visible variables directly in the wrapper template. Convert each variable to serializable `{label, display_value, type}` objects from `GlobalServiceCatalogUtil().getVariablesForTask(...)`; do not expose the original variable objects to Angular, because the embedded widget/option serialization can drop the child content even when the wrapper has the data.
- For service-specific custom tabs on a broad `sysapproval` approval task config, prefer guarding inside the custom widget rather than using tab `visibility_script` for the HR case lookup. PDI portal testing showed a tab-level scripted check can be suppressed in the Employee Center runtime/cross-scope path, hiding the tab entirely. Keep the custom tab visible, pass the approval sys_id, and let the widget render nothing unless the resolved HR case has the target `hr_service`.
- Do not use encoded-query dot-walks such as `sysapproval.hr_service!=...` as proof of service filtering; PDI testing showed those can match incorrectly even when a direct GlideRecord case lookup returns the opposite result.
- The tab configuration `table` field is read-only and can remain blank on API-created tab rows. `sn_ex_sp.ApprovalHubUtilSNC.getSecondaryInfo()` uses the task config's resolved parent/reference record to derive `tabObj.parentTable` and `tabObj.sysId`; it does not rely on the tab row's `table` value for rendering.
- Variable Summarizer displays submitted variable/question data for the target record. For HRSD approvals, verify the HR case was created through the portal/record producer path and has question-answer data; backend-created cases may not show variables even when the tab configuration is correct.
- For reusable HR-service-specific approval body text, use a small config table such as `u_hr_approval_todo_content` in Global with `u_active`, `u_hr_service`, `u_body`, `u_show_variables`, and `u_order`, then keep one Employee Center Core custom tab/widget that resolves `sysapproval_approver -> HR case`, looks up the active config row by `hr_service`, renders the configured text, and renders serialized variables from `global.GlobalServiceCatalogUtil().getVariablesForTask(...)`. Put the table/data in a Global update set and the widget/task/tab records in an Employee Center Core update set. If the config row itself must move in the update set, set the custom table collection dictionary attribute `update_synch=true` and save the row with `GlideUpdateManager2`.
- When HR approval body text needs links or basic formatting, add a separate HTML field such as `u_body_html` instead of changing the existing plain string `u_body`. Have the widget prefer `u_body_html`, fall back to `u_body`, render the rich value with `ng-bind-html` plus `$sce.trustAsHtml` only for trusted administrator-maintained content, and use a link function to force rendered anchors to `target="_blank"` with `rel="noopener noreferrer"`. Keep variables rendered separately below the intro.
- `sn_ex_sp_task_configuration.name` is `translated_text`; after renaming it through Table API or GlideRecord, verify both the task record and any `sys_translated_text` row for `tablename=sn_ex_sp_task_configuration`, `fieldname=name`, and the target `documentkey`. A stale language row can make UI/Table API reads show the old name even when the base value and customer-update target name are correct.
