# Integration and Import Lessons

- For SAP SuccessFactors, check whether the ServiceNow SuccessFactors Spoke is licensed and available before designing a custom REST integration. It provides connection aliases, sample flows/subflows/actions, staging tables, transform maps, and webhook processing.
- SAP SuccessFactors Basic Auth is deprecated. Prefer OAuth/OIDC designs, usually OAuth 2.0 with SAML bearer and certificate/JKS setup when using the spoke pattern.
- Confirm SuccessFactors API server, Company ID, API family/version, entity model, permissions, IP restrictions, delta strategy, and pagination strategy before building ServiceNow records.
- Integration Center and Data Model Navigator on the SAP side are useful discovery tools for fields, relationships, and simple exports. Some OData V4 scenarios need SAP Cloud Integration or Business Accelerator Hub packages instead.
- Integration testing is layered: use REST API Explorer to discover and first-test inbound ServiceNow REST APIs, Postman for repeatable external-client and vendor-contract requests, and ServiceNow Table API helpers for metadata, logs, target state, imports, update capture, and narrow disposable setup.
- A Postman success does not prove the ServiceNow outbound runtime path. After vendor contract discovery, prove transport again from ServiceNow with the chosen Spoke/IntegrationHub/REST Message path and inspect `sys_outbound_http_log`.
- A REST API Explorer or Postman inbound success does not prove ServiceNow side effects. Read the target/staging/log records through Table API helpers and test validation, auth, idempotency, and errors separately.
- The PDI has REST API Explorer available at `/$restapi.do` and Postman MCP access to the personal `ServiceNow integration` workspace with a `ServiceNow` collection as of 2026-05-22.
- For Compendia HR knowledge, get the standard API contract and source-of-truth/update rules first. Preserve sanitized vendor examples in Postman, then verify ServiceNow knowledge staging/target state and source identifiers through Table API reads.
- For Norwegian postcode/place validation, use Posten/Bring's official free postcode register before third-party APIs: `https://www.bring.no/postnummerregister-ansi.txt`. Layout is tab-separated ANSI with `postnummer`, `poststed`, `kommunekode`, `kommunenavn`, and `kategori`.
- Convert the Bring register from Windows-1252/ANSI to UTF-8 before upload. A direct `RESTMessageV2` fetch from the PDI can return replacement characters for `Æ`, `Ø`, and `Å`; uploading a UTF-8 attachment and parsing with `GlideTextReader` preserved values.
- Avoid one Table API write per postcode row for full-register imports. Uploading the TSV as a story attachment and running a bounded Xplore import completed quickly for the 5,122-row Bring file.
- For Unit4-style validation, store at least `u_postal_code`, `u_city`, and `u_normalized_city`. Keeping `u_municipality_code`, `u_municipality_name`, `u_category`, `u_source`, `u_active`, and `u_last_imported` improves diagnostics without storing sensitive user data.
- Import Set demo validated in the PDI on 2026-05-14: a File/CSV Data Source with Attachment retrieval loaded through `GlideImportSetLoader`, transformed through `GlideImportSetTransformer`, and reported through `sys_import_set_run` plus staging row states. Coalesce on a unique external ID prevented duplicates; unchanged coalesced rows were reported as ignored with "No field values changed", while one deliberately changed target row produced `updates=1`.
- For reference imports, setting the Field Map `choice_action` to `ignore` avoided creating a bad `sys_user` reference when the source owner display value did not resolve. For choice/date cleanup, small Field Map source scripts were enough to normalize status aliases and `MM/dd/yyyy` dates before transform.
